# DOD_GUIDE.md — DoR / DoD 작성 가이드 + Phase 아키타입

## 1. Definition of Ready (DoR) — 진입조건
phase를 시작하기 전 만족해야 하는 것. 충족 안 되면 시작 금지.
- 선행 phase의 DoD 게이트가 통과되어 있다.
- 필요한 환경·자원·접근이 준비되어 있다(`docs/ENVIRONMENT`).
- 관련 백서 절(`docs/`)과 불변식(`INVARIANTS.md`)을 확인했다.
- 입출력 계약/스키마가 정해져 있다(해당 시).

## 2. DoD 작성 원칙
- **측정 가능해야 한다.** 사람이 "잘 된 것 같다"가 아니라 명령 출력·수치·통과/실패 이진값으로 확인.
- **"코드가 돈다" != "기능이 동작한다".** 효과를 직접 보는 게이트를 둔다(성공률·정확도·왕복 일치).
- **불변식은 분리해 명시하고 INV-n로 인용.** 1건 위반 시 통과 금지.
- **정합성(parity) 게이트.** 같은 로직을 두 곳에서 구현하면 동일 입력→동일 출력을 수치로 검증.
- **재현 가능한 명령으로 검증.** 누가 언제 돌려도 같은 판정.
- **증거를 남긴다.** 통과 주장 시 명령·핵심 출력을 phase 파일/`PROGRESS.md`에.

## 3. Phase 아키타입 메뉴 (필요한 것만 골라 `phases/Mn.md`로)
1. **기반/스캐폴딩** — 레포·패키지·빌드/린트/타입 설정·경계 룰. DoD: 빌드/린트/타입체크 그린, 경계 위반 lint 차단(INV-3).
2. **핵심 도메인 로직** — 부수효과 없는 순수 로직. DoD: 단위 테스트 그린, 결정론, 알려진 케이스 일치.
3. **데이터/영속화** — 스키마·마이그레이션·저장소. DoD: 마이그레이션 성공, CRUD 왕복, 스키마-타입 정합.
4. **API/서버** — 엔드포인트·계약·헬스. DoD: 기동+헬스 200, 핵심 엔드포인트 왕복, 스키마 검증.
5. **핵심 기능 실효 검증** ★ — 실제로 의도한 결과를 내는지. DoD: 목표 지표 기준선 도달, 발산/예외 없음, 산출물 저장·재로드 재현.
6. **통합/정합성** ★ — 두 구현/환경 일치. DoD: 고정 입력 셋 양측 출력 일치(INV-5).
7. **UI/프론트엔드** — 화면·상태·실시간. DoD: 빌드 그린, 핵심 플로우 수동 시연, 디자인 토큰, 가로 스크롤 없음 + 기본 접근성.
8. **외부 연동** — 서드파티·확장·OS. DoD: 실제 대상 동작 시연, 변경 시 계약 파일 한 곳만, 가드레일 준수.
9. **벤치마크/배포/CI** — 측정·파이프라인·산출물. DoD: CI 그린, 벤치 리포트, 배포 아티팩트.

> 추상 "예시 DoD"를 프로젝트의 구체적 수치·명령으로 바꾼다. (예: "목표 지표 도달" → "Beginner 승률 0.85 도달".)
