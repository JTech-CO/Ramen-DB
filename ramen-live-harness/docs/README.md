# docs/ — 백서 세트 (무엇을 만드는가)

> 하네스가 "어떻게"라면, `docs/`는 "무엇을". 각 phase는 작업 전 관련 백서 절을 읽는다. 분량이 커지면 절별로 파일을 분리한다(`docs/TECHNICAL.md` 등). 아래는 최소 골격.

## 작성할 문서
- **기술 백서(TECHNICAL)** — 아키텍처·데이터 흐름·핵심 알고리즘·인터페이스 계약.
- **설계 백서(DESIGN)** — UI/UX, 디자인 토큰, 상태·화면 흐름, 접근성 기준.
- **파일 트리(FILE_TREE)** — 디렉터리/패키지 구조와 **모듈 경계 규칙**(누가 누구를 import 가능한가). INV-3의 근거.
- **환경(ENVIRONMENT)** — 로컬·CI 셋업, 런타임/의존성 버전, 시크릿·환경변수, 실행/검증 명령.

## TECHNICAL 골격
1. 목표·범위 / 2. 시스템 구성 / 3. 데이터 모델·스키마 / 4. 핵심 알고리즘·로직 / 5. 인터페이스·계약(입출력) / 6. 정합성이 필요한 지점(INV-5) / 7. 성능·확장 고려.

## DESIGN 골격
1. 원칙·톤 / 2. 디자인 토큰(색·타이포·간격) / 3. 화면·상태 흐름 / 4. 컴포넌트 / 5. 접근성(키보드·aria·색 외 표식) / 6. 금지 패턴(anti-cliché).

## FILE_TREE 골격
1. 디렉터리 트리 / 2. 패키지별 책임 / 3. **경계 규칙 표**(allow/deny import) / 4. 공유 계층 정의.

## ENVIRONMENT 골격
1. 사전 요구(런타임·도구 버전) / 2. 설치 절차 / 3. 환경변수·시크릿(.env 키 목록, 값은 비움) / 4. 실행 명령 / 5. 검증 명령(테스트·린트·빌드) / 6. 자원 요건(해당 시).
