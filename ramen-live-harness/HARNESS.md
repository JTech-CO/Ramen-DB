# HARNESS.md — 마스터 운영 매뉴얼

> "무엇을 만드는가"(백서)가 아니라 **"어떻게 진행·검증·복구하는가"(운영 규율)**. phase의 "완료"는 코드가 도는 것이 아니라 **측정 가능한 게이트(DoD) 통과**다. 게이트로 막지 않으면 잘못된 완료 판정이 다음 phase로 전파된다.

## 0. 세션 루프
1. 시작: `PROGRESS.md` → 현재 phase·다음 할 일·미결 → 해당 `phases/Mn.md` + 관련 `docs/` 절.
2. 작업: 한 번에 한 phase. 작업 단위마다 그 phase의 검증 명령으로 게이트 확인.
3. 종료: `PROGRESS.md` 갱신(완료 체크·증거·다음 할 일·새 미결·결정 링크) → 커밋.

### 0.1 완료 판정
DoD 항목이 **전부** 충족되어야 완료. 하나라도 미달이면 미완료로 두고 다음 phase로 가지 않는다.

### 0.2 의존 순서
M0(스캐폴딩) → M1(Tier1 인제스트·조인 ★) → M2(단종 status 도출 ★) → M3(주간 스냅샷 파이프라인). 여기까지가 제품 Phase 0(가공라면 DB)의 빌드 경로다.
이후 로드맵: M4(음식점 레이어=LOCALDATA) → M5(Tier2 실시간 조회·제휴 ★) → M6(UI/프론트엔드) → M7(UGC 레시피·맛집 큐레이션). M4 이후는 도달 시 `phases/_TEMPLATE.md`로 인스턴스화한다.
각 phase의 진입조건(DoR)이 선행 게이트를 명시.

### 0.3 증거 기반 완료
게이트 "통과"를 주장할 때, 근거가 되는 **명령과 핵심 출력**(테스트 요약·수치·해시·diff)을 해당 `phases/Mn.md`의 증거 칸과 `PROGRESS.md`에 남긴다. 증거 없는 완료 주장은 미완료로 본다.

## 1. 파일 라우팅
- phase 상세·DoD·검증: `phases/Mn.md` (템플릿: `phases/_TEMPLATE.md`)
- DoD/DoR 작성법·아키타입: `gates/DOD_GUIDE.md`
- 불변식: `INVARIANTS.md` (DoD·레드라인이 INV-n로 참조)
- 막혔을 때: `RUNBOOK.md`
- 결정 기록: `decisions/`
- 무엇을 만드는가: `docs/` (특히 `TECHNICAL.md`, 경계는 `FILE_TREE.md`)

## 2. 멈춤 규칙 (STOP)
### 2.1 멈춰야 하는 상황
- 같은 에러/실패를 서로 다른 방법으로 **3회** 시도해도 미해결.
- DoD 게이트(특히 불변식·핵심 기능 실효·정합성)를 못 넘는데 우회 충동.
- 불변식(`INVARIANTS.md`)을 깨야만 통과 가능.
- 큰 아키텍처 변경 필요 추정.
- 외부 제약(공공데이터 접근 불가·상용 API 약관/정책 위반 소지).
### 2.2 멈출 때 절차
1. `PROGRESS.md`에 기록: 증상 / 재현 방법 / 시도한 것들 / 가설 / 막힌 지점.
2. 사용자에게 요약 보고, 선택지가 있으면 제시한 뒤 결정 요청.
3. 결정 전까지 불변식을 깨는 임시 우회를 만들지 않는다.
### 2.3 절대 금지 (레드라인)
`CLAUDE.md`의 레드라인과 동일. 요지: Tier2 영속화(INV-4)·status 단정(INV-6)·크롤링(INV-8)·PK 가변(INV-5)·테스트/게이트 위장·불변식 깨고 진행·경계 임의 변경·시크릿/대용량 커밋·가드레일 약화.

## 3. 검증 우선순위 (한 줄)
불변식/규칙 정확성(특히 INV-4 비영속·INV-6 status·INV-5 PK) > 핵심 기능 실효(인제스트·조인·단종 도출 정확성) > 통합 정합성(스냅샷 idempotency INV-7) > UX/배포.
앞 단계 게이트가 깨지면 뒤 단계는 의미 없으므로 항상 앞에서부터 굳힌다.
