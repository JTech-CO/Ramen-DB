# ADR-0006: 스냅샷 산출 포맷은 JSON, 배포는 GitHub Actions 아티팩트 + 정적 호스팅(MVP)

- **상태**: 채택 (MVP, 재검토 가능)
- **일시**: 2026-06-18
- **관련**: phase M3, INV-7·INV-2, Q4, `docs/TECHNICAL` §3.1, ADR-0001

## 맥락
Q4(호스팅·산출 포맷)는 M3 빌드를 위해 결정이 필요하다. 백서 §3.1은 "정적 산출물(JSON/SQLite)"과 "CDN/정적 또는 Pi"를 후보로 둔다. Tier1 데이터량은 작아(식품 수만·음식점 수십만) 상시 서버가 필수는 아니다.

## 결정
- **산출 포맷**: **JSON**(키 정렬 결정론 직렬화). 이유: idempotent 해시(INV-7)와 정적 호스팅에 가장 단순하고, diff·검수가 쉽다. 본문(`<version>.json` = version+products)만 해시 대상으로 하고 생성시각·리포트는 `<version>.meta.json`으로 분리(INV-7, RUNBOOK 4).
- **배포**: GitHub Actions cron(주간) → `actions/upload-artifact`로 산출. 정적/CDN 호스팅에 동일 JSON을 게시. 산출물은 레포에 커밋하지 않는다(`snapshots/` → .gitignore, INV-2).
- **서비스키**: Actions repository secret로만 주입(INV-1). 미설정 시 샘플 데이터로 산출(데모/CI).

## 근거
JSON은 결정론 직렬화·해시 검증·정적 배포가 즉시 가능하고 외부 의존이 없다. 데이터량이 작아 SQLite의 쿼리 이점이 아직 필요 없다. 정적 산출+CDN은 상시 서버 없이 운영 가능하다.

## 결과 / 트레이드오프
- 이득: INV-7 검증(해시 일치)·정적 배포·diff가 단순. 호스팅 비용 최소.
- 비용: 대용량·복합 질의가 필요해지면(예: 음식점 수십만 + 지도 질의, M4 이후) JSON 단일 파일은 비효율 → 그때 SQLite/인덱스 산출을 추가 검토. 포맷 전환은 pipeline의 직렬화 레이어 교체로 국소화된다(가역적).

## 검토한 대안
- **SQLite 산출** — 질의·인덱스 이점이 있으나 현 데이터량엔 과함, 해시·정적배포가 더 번거로움. M4 이후 재검토.
- **Raspberry Pi 상시 서버** — 정적 산출로 충분한 현 단계엔 운영 부담만 큼. 보류.

> 호스팅 형태(특정 CDN/도메인)·SQLite 전환은 운영자 재검토 대상으로 열어 둔다(Q4 잔여).
